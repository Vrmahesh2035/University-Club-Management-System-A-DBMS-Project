Database Normalization

How this schema satisfies 1NF, 2NF, and 3NF

1NF

First Normal Form — Atomic Values

A relation is in 1NF if every attribute contains only atomic (indivisible) values and every row is unique.

✓ No repeating groups or arrays
✓ Each column holds a single value
✓ Every table has a Primary Key
✓ No multi-valued attributes

Applied in this schema:

Table1NF Compliance
students Each student has ONE phone number (atomic), ONE email. No list of phones stored in one cell. Student-club relationships moved to memberships.
memberships Each row = one (student, club) pair. A student in 3 clubs = 3 rows — not one row with a list of clubs.
event_registrations Each registration is a separate atomic row with its own attendance_status. No comma-separated student lists.
club_advisor_assignments Each advisor–club pair is a separate row. A club with 2 advisors = 2 rows.
✗ Violates 1NF
students: (id=1, name="Arjun", clubs="Code Crafters, Debate Society")
✓ Satisfies 1NF
memberships: (student_id=1, club_id=1)
memberships: (student_id=1, club_id=7)
2NF

Second Normal Form — Full Functional Dependency

A relation is in 2NF if it is in 1NF AND every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies).

✓ Must already be in 1NF
✓ No partial dependencies on composite keys
✓ Each non-key attribute depends on the FULL PK

Composite Keys and Partial Dependency Analysis:

TableComposite Key2NF Compliance
memberships (student_id, club_id) join_date depends on BOTH (when did THIS student join THIS club). position_id depends on BOTH (a student can be President in one club, Member in another). status depends on BOTH. ✓ No partial deps.
event_registrations (event_id, student_id) registration_date depends on BOTH (when did THIS student register for THIS event). attendance_status depends on BOTH. ✓ No partial deps.
club_advisor_assignments (club_id, advisor_id) assigned_date depends on BOTH. ✓ No partial deps.
✗ Violates 2NF (partial dependency)
memberships(student_id, club_id, student_name, club_budget)
— student_name depends only on student_id (partial dep!)
— club_budget depends only on club_id (partial dep!)
✓ Satisfies 2NF
memberships(student_id, club_id, position_id, join_date, status)
— All attributes depend on the FULL composite key.
3NF

Third Normal Form — No Transitive Dependencies

A relation is in 3NF if it is in 2NF AND no non-key attribute is transitively dependent on the primary key through another non-key attribute.

✓ Must already be in 2NF
✓ No transitive dependencies (A→B→C where A is PK)
✓ Non-key attributes depend ONLY on the primary key

Transitive Dependencies Removed:

Transitive Chain EliminatedHow
student_id → dept_id → dept_name, faculty_dean dept_name and faculty_dean are in the departments table. students stores only dept_id (FK). No transitive dep in students.
club_id → category_id → category_name, description Club category metadata lives in club_categories. clubs stores only category_id (FK).
membership_id → position_id → position_name, description Position metadata lives in positions. memberships stores only position_id (FK).
event_id → venue_id → capacity, location Venue details live in venues. events stores only venue_id (FK). No transitive dep.
advisor_id → dept_id → dept_name Advisor's department is in departments. advisors stores only dept_id.
✗ Violates 3NF (transitive dependency)
events(event_id, event_name, venue_id, venue_name, venue_capacity)
— event_id → venue_id → venue_name (transitive!)
— event_id → venue_id → venue_capacity (transitive!)
✓ Satisfies 3NF
events(event_id, event_name, venue_id, …)
venues(venue_id, venue_name, capacity, location)
— venue_name and capacity depend directly on venue_id (their own PK).
